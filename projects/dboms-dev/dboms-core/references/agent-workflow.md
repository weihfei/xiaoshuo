## 1. 智能体角色

DBOMS 项目采用基于 OpenSkills 框架的智能体协作系统，定义了三个核心角色：

### 1.1 项目经理（Project Manager）

**技能名称**: `project-manager`  
**职责**: 需求分析、任务拆分、进度跟踪、质量把控

### 1.2 程序员（Developer）

**技能名称**: `developer`  
**职责**: 代码实现、规范遵循、单元测试、技术文档

### 1.3 测试员（Tester）

**技能名称**: `tester`  
**职责**: 测试用例设计、执行测试、缺陷管理、测试报告

## 2. 角色职责详解

### 2.1 项目经理职责

#### 核心能力
- **需求分析**: 理解用户需求，识别关键功能点
- **任务拆分**: 将复杂需求拆分为可执行的小任务
- **资源分配**: 根据任务类型分配给对应角色
- **进度跟踪**: 监控任务完成情况，确保按时交付
- **质量审核**: 最终成果验收，确保符合规范

#### 工作流程
```
1. 接收用户需求
   ↓
2. 分析功能点和技术难点
   ↓
3. 创建任务列表（使用 add_tasks）
   ↓
4. 分配任务给程序员/测试员
   ↓
5. 跟踪任务进度
   ↓
6. 组织代码审查
   ↓
7. 验收交付成果
```

#### 典型场景
```
用户：我想新增一个合同管理功能

项目经理：
1. 分析需求：合同增删改查、审批流程、统计报表
2. 拆分任务：
   - 创建 API接口定义
   - 开发合同列表页面
   - 开发合同详情页面
   - 开发合同编辑表单
   - 编写单元测试
   - 编写 E2E 测试
3. 分配任务给对应角色
4. 设定优先级和时间表
```

### 2.2 程序员职责

#### 核心能力
- **代码实现**: 按照需求和技术规范编写高质量代码
- **规范遵循**: 严格遵守项目代码规范（命名、结构、注释等）
- **单元测试**: 为关键业务逻辑编写单元测试
- **代码优化**: 持续改进代码质量和性能
- **技术文档**: 编写函数注释、组件说明等技术文档

#### 工作流程
```
1. 接收任务（来自项目经理）
   ↓
2. 理解需求和设计规范
   ↓
3. 技术方案设计
   ↓
4. 编写代码（遵循 dboms-core 规范）
   ↓
5. 自测（运行单元测试）
   ↓
6. 代码审查准备
   ↓
7. 提交审查
```

#### 编码规范要点
- **命名规范**: 
  - 业务组件：`Db` + 大驼峰（如 `DbContractDetail`）
  - 基础组件：`E` + 大驼峰（如 `EButton`）
  - 模板使用：kebab-case（如 `<e-button>`）
  
- **代码组织**:
  - 接口定义在 `src/interface/`
  - API模块化在 `src/api/urls/modules/`
  - Store 模块化在 `src/store/modules/`

- **组件规范**:
  - 使用 `<script setup lang="ts">`
  - 单个组件不超过 500 行
  - Props 详细定义类型和验证

- **TypeScript**:
  - 禁止使用 `any`
  - 接口使用 `I` 前缀
  - 类型别名使用 `T` 前缀

#### 典型场景
```
任务：开发合同列表页面

程序员：
1. 查看设计规范
2. 创建文件：src/views/contract/contractList/index.vue
3. 按规范组织代码：
   - 导入顺序：Vue → 第三方库 → 项目配置 → Store → 组件 → API → 类型
   - 响应式数据：ref/reactive
   - 计算属性：computed
   - 方法定义：明确参数和返回类型
4. 使用 ELESUI组件
5. 编写组件注释
6. 运行单元测试
```

### 2.3 测试员职责

#### 核心能力
- **测试用例设计**: 使用等价类划分、边界值分析等方法
- **单元测试**: 使用 Vitest 编写和执行单元测试
- **E2E 测试**: 使用 Playwright 进行端到端测试
- **缺陷管理**: 记录、跟踪和验证 bug 修复
- **测试报告**: 生成详细的测试结果报告

#### 工作流程
```
1. 审查需求文档
   ↓
2. 设计测试用例
   ↓
3. 编写测试代码
   ↓
4. 执行测试
   ↓
5. 提交缺陷报告
   ↓
6. 验证修复
   ↓
7. 生成测试报告
```

#### 测试方法
- **等价类划分**: 将输入数据分为有效/无效类别
- **边界值分析**: 测试极限值（最大值、最小值、空值等）
- **场景法**: 模拟真实用户操作流程
- **错误推测**: 基于经验预测可能出现的问题

#### 典型场景
```
功能：合同搜索功能

测试员：
1. 设计测试用例：
   - 有效输入：正常关键词搜索
   - 边界值：空字符串、超长字符串、特殊字符
   - 场景：搜索后分页、排序、筛选
   
2. 编写测试代码：
   - 单元测试：搜索逻辑、过滤函数
   - E2E 测试：完整搜索流程
   
3. 执行测试并生成报告
```

## 4. 自动化调度机制

### 4.1 监督者模式

Node.js 服务作为监督者，负责：
- **智能路由**: 根据请求类型自动路由到对应 skill
- **上下文管理**: 维护对话历史和任务状态
- **结果聚合**: 汇总多个技能的输出
- **异常处理**: 处理错误和重试

### 4.2 自动触发规则

| 触发条件 | 自动调用的 Skill |
|---------|-----------------|
| 新功能开发需求 | project-manager → developer |
| 代码审查请求 | ebvue3-frontend-code-review |
| 测试相关请求 | tester |
| Bug 修复需求 | developer → tester |
| 文档编写需求 | doc-coauthoring |

### 4.3 上下文传递

```typescript
// 示例：上下文传递链
const context = {
  requestId: 'req-001',
  userQuery: '新增合同管理功能',
  tasks: [
    { id: 't1', type: 'analysis', assignedTo: 'pm' },
    { id: 't2', type: 'implementation', assignedTo: 'dev' },
    { id: 't3', type: 'testing', assignedTo: 'tester' }
  ],
  sharedState: {
    requirements: {...},
    design: {...},
    code: {...},
    tests: {...}
  }
}
```

## 5. 技能封装规范

### 5.1 Skill 结构

```
.claude/skills/{skill-name}/
├── SKILL.md              # 技能主文件（必需）
├── references/           # 参考资料（可选）
│   ├── standards.md
│   └── examples.md
├── templates/            # 模板文件（可选）
│   └── component.vue
└── scripts/              # 辅助脚本（可选）
    └── validate.ts
```

### 5.2 SKILL.md 格式

```markdown
---
name: skill-name
description: 技能描述
license: MIT
metadata:
  author: 作者名
  version: "1.0"
  project: dboms
allowed-tools: "Bash(*) Read Write"
---

# 技能说明

## 核心功能
- 功能 1
- 功能 2

## 使用场景
- 场景 1
- 场景 2

# 使用方法

## 常用命令
command 1
command 2

# 注意事项
...
```

### 5.3 本地化集成

**优势**:
- ✅ 减少网络请求，提高响应速度
- ✅ 版本可控，便于团队协作
- ✅ 离线可用，不依赖外部服务

**步骤**:
```bash
# 1. 创建 skill 目录
mkdir -p .claude/skills/dboms-core

# 2. 创建 SKILL.md
touch .claude/skills/dboms-core/SKILL.md

# 3. 添加参考文档
mkdir -p .claude/skills/dboms-core/references

# 4. 读取 skill
npx openskills read dboms-core
```

## 6. 最佳实践

### 6.1 任务拆分原则

**SMART 原则**:
- **S**pecific: 具体明确
- **M**easurable: 可衡量
- **A**chievable: 可实现
- **R**elevant: 相关性
- **T**ime-bound: 时限性

**示例**:
```
❌ 错误：开发合同管理功能（太大）

✅ 正确拆分：
- 创建合同列表 API（2h）
- 开发合同列表页面（4h）
- 开发合同详情页面（3h）
- 开发合同编辑表单（4h）
- 编写列表页单元测试（2h）
- 编写详情页 E2E 测试（2h）
```

### 6.2 代码审查清单

**开发者自检**:
- [ ] 组件命名符合规范
- [ ] 代码结构符合要求
- [ ] TypeScript 类型完整
- [ ] 注释密度适当（10-20%）
- [ ] console.log ≤ 3 个
- [ ] 组件行数 < 500
- [ ] 通过 ESLint/Prettier
- [ ] 单元测试通过

**审查者检查**:
- [ ] 功能完整性
- [ ] 规范遵循度
- [ ] 代码质量
- [ ] 性能优化
- [ ] 安全性
- [ ] 测试覆盖率

### 6.3 沟通协作技巧

**清晰的需求描述**:
```
背景：当前合同管理功能不完善
目标：新增合同审批流程和统计报表
具体要求：
1. 支持多级审批
2. 支持自定义审批人
3. 统计报表包含月度/季度/年度
4. 下周五前完成
```

**有效的反馈**:
```
✅ 好：第 35 行的变量命名不符合规范，建议改为 camelCase

❌ 差：这里写得不好
```

### 6.4 性能优化建议

**智能体协作优化**:
- 减少不必要的角色切换
- 批量处理相似任务
- 共享中间结果
- 合理使用缓存

**代码执行优化**:
- 并行执行独立任务
- 懒加载大型技能
- 预加载常用技能
- 定期清理过期缓存

## 7. 常见问题

### Q1: 如何选择合适的 Skill？

**A**: 根据任务类型选择：
- 需求分析 → project-manager
- 代码开发 → developer
- 测试相关 → tester
- 代码审查 → ebvue3-frontend-code-review
- 文档编写 → doc-coauthoring

### Q2: 多个 Skill 如何协作？

**A**: 通过监督者服务自动调度：
1. 项目经理分析需求并创建任务
2. 程序员实现具体功能
3. 测试员验证质量
4. 项目经理最终验收

### Q3: 如何保证 Skill 的输出质量？

**A**: 
- 遵循标准化文档结构
- 提供详细的参考规范
- 建立审查机制
- 持续更新和优化
